## 第1章 メモリマップとレジスタ

### 1.1 コンピュータの内部構造の再確認

まず、一般的なコンピュータ（PCやマイコンを含む）の内部構造を簡単に整理する。

* CPU（中央処理装置）：命令を読み取り、演算を行う。
* メモリ（RAM, ROM）：データやプログラムを格納する。
* 入出力回路：外部デバイスとやり取りする。

概念図は以下のようになる。

```text
           ┌──────────────┐
           │      CPU     │
           │ (演算＋制御)  │
           └─────┬────────┘
                 │
       ┌─────────┴──────────┐
       │                    │
┌─────────────┐      ┌──────────────┐
│   メモリ     │     │ 入出力回路     │
│ (RAM/ROM)   │      │ (GPIO, UART…)│
└─────────────┘      └──────────────┘
```

CPUは、**命令（Instruction）**と**データ**を扱う。
命令はメモリから読み出され、データも同様にメモリから読み出される。
このとき、CPU内部では「レジスタ」と呼ばれる非常に高速な記憶領域が使われる。

---

### 1.2 CPUレジスタとは何か

レジスタは、CPU内部にある小さなメモリである。容量はごくわずかだが、
メインメモリ（RAM）より桁違いに高速でアクセスできる。

#### メモリ階層のイメージ

| 場所     | 典型容量     | アクセス速度 | 例                |
| ------ | -------- | ------ | ---------------- |
| レジスタ   | 数十〜数百バイト | 非常に高速  | R0, R1, PC, SP 等 |
| キャッシュ  | KB〜MB    | 高速     | L1, L2キャッシュ      |
| メインメモリ | 数GB      | 中程度    | DRAM             |
| ストレージ  | 数百GB〜TB  | 低速     | SSD, HDD         |

C言語で

```c
int a = 1, b = 2, c;
c = a + b;
```

という式を評価するとき、概念的には以下のような流れになる。

1. メモリ上の `a` の値(1)がレジスタ（例えば R0）にロードされる。
2. メモリ上の `b` の値(2)が別のレジスタ（例えば R1）にロードされる。
3. `R2 = R0 + R1` のようにレジスタ間で演算が行われる。
4. 結果が再びメモリ上の `c` の領域にストアされる。

```text
メモリ(a) ──► R0
メモリ(b) ──► R1
           R2 = R0 + R1
R2 ──► メモリ(c)
```

このように、レジスタは**計算の一時置き場**として使われている。

---

### 1.3 メモリマップとメモリマップドI/O

次に、**メモリマップ**という概念を導入する。
CPUは「アドレス」という番号を指定してメモリにアクセスする。
このアドレス空間に、「普通のメモリ」と「周辺回路（入出力）」が共存している場合がある。
これを**メモリマップドI/O**と呼ぶ。

例として、マイコン内部のアドレス空間を非常に単純化して示す。

| アドレス範囲                  | 用途             |
| ----------------------- | -------------- |
| 0x0000_0000〜0x0000_3FFF | RAM（変数など）      |
| 0x0000_4000〜0x0000_FFFF | ROM（プログラム）     |
| 0x1000_0000〜0x1000_00FF | GPIOレジスタ群      |
| 0x1000_0100〜0x1000_01FF | I2Cコントローラレジスタ群 |
| 0x1000_0200〜0x1000_02FF | SPIコントローラレジスタ群 |

CPUから見ると、どれも「アドレス」でしかない。
しかしその実態は、ある範囲がRAM、ある範囲はGPIOレジスタ、というように機能が分かれている。
この「アドレスと機能の対応表」が**メモリマップ**である。

---

### 1.4 組込みにおけるレジスタ利用：GPIOの制御

組込み開発でよく出てくる例として、GPIO（汎用入出力）の制御を考える。

1本のGPIOピンは、内部の**出力レジスタ**の1ビットに対応していることが多い。
例えば8ビットの `PORTB` レジスタのビット5がLED付きピンPB5に対応しているとしよう。

```c
#define PORTB (*(volatile uint8_t*)0x10000000)

PORTB |=  (1 << 5);  // PB5 を High 出力にする
PORTB &= ~(1 << 5);  // PB5 を Low 出力にする
```

ここで起きていることは、

* `PORTB` というレジスタのビット5を1にすると、PB5ピンがHigh電位になる
* ビット5を0にすると、PB5ピンがLow電位になる

という非常に単純な対応である。
`digitalWrite(13, HIGH);` のようなArduinoの関数は、**内部でこのようなレジスタ操作を行っている**だけである。

---

#### 小まとめ（第1章 前半）

* CPUレジスタは「計算用の高速な引き出し」である。
* メモリマップとは、アドレス空間に「RAM」「ROM」「周辺レジスタ」などが割り当てられた対応表である。
* 組込みでは、特定のアドレスにあるレジスタを書き換えることで、GPIOピンの電圧を変更できる。

---

### 1.5 センサ内部のレジスタ：BME280を例に

ここからは、温湿度気圧センサ **BME280** を例に、
「センサのレジスタを叩いて値を得る」流れを概念的に説明する。

BME280の内部構造を非常に簡略化すると以下のようになる。

```text
   温度・湿度・気圧
         │
     [ センサ素子 ]
         │  (アナログ信号)
     [ A/D変換 ]
         │   (デジタル)
     [ レジスタ群 ]
      │    │    │
   設定  状態  データ
```

この「レジスタ群」の内容は、データシートに**レジスタマップ**として記載されている。

以下は説明用に簡略化したレジスタマップの例である（実際のBME280とは異なる可能性があることに注意）。

| アドレス    | レジスタ名         | 役割              |
| ------- | ------------- | --------------- |
| 0xD0    | `ID`          | チップID確認         |
| 0xF2    | `CTRL_HUM`    | 湿度オーバサンプリング設定   |
| 0xF4    | `CTRL_MEAS`   | 温度・気圧の測定モード設定   |
| 0xF3    | `STATUS`      | 測定中フラグなど        |
| 0xF7〜F9 | `PRESS_MSB..` | 気圧生データ(20bit程度) |
| 0xFA〜FC | `TEMP_MSB..`  | 温度生データ          |
| 0xFD〜FF | `HUM_MSB..`   | 湿度生データ          |

センサから温度データを得る流れは概ね次のようになる。

1. `ID` レジスタを読み、正しいチップか確認する。
2. `CTRL_HUM` や `CTRL_MEAS` に値を書き込み、測定モードを設定する。
3. `STATUS` を読み、測定完了を待つ。
4. `TEMP_MSB`〜`TEMP_XLSB` を読み出し、生データを取得する。
5. 生データに補正式を適用し、[℃]に変換する。

---

### 1.6 ビットシフトと物理量の取得

BME280の温度データは、複数バイトに分かれて格納されている。
簡略化した例として、温度データが20bit幅で `TEMP_MSB`（上位8bit）、`TEMP_LSB`（中間8bit）、`TEMP_XLSB`（下位4bit）に格納されているとする。

このとき、生データ `raw_temp` は次のように組み立てられる。

```c
uint8_t msb  = read_reg(0xFA); // TEMP_MSB
uint8_t lsb  = read_reg(0xFB); // TEMP_LSB
uint8_t xlsb = read_reg(0xFC); // TEMP_XLSB

int32_t raw_temp = ( (int32_t)msb  << 12 )
                 | ( (int32_t)lsb  << 4  )
                 | ( (int32_t)xlsb >> 4  );
```

ここで行っていることは、

* `msb` を12ビット左シフト → 上位8ビットを20ビット幅の上位8ビット位置へ
* `lsb` を4ビット左シフト  → 中間8ビット位置へ
* `xlsb` の上位4ビットを右シフトで取り出して下位4ビットへ

というビット操作である。

実際のBME280では、さらに較正用のパラメータを用いた補正計算が必要であるが、
ここでは単純な例として

[
T[℃] = \frac{\text{raw_temp}}{512.0}
]

のような線形関係を仮定しよう。

```c
float temp_c = raw_temp / 512.0f;
```

このように、
**「レジスタからバイト列を読み出す → ビット操作で1つの整数値にする → 浮動小数点演算などで物理量に変換する」**
という流れでセンサ値が得られる。

---

### 1.7 ドライバという抽象化

ここまでの流れを、そのままアプリケーションコード内に毎回書いていると、非常に煩雑になる。

* 毎回レジスタアドレスを書く
* 毎回ビットシフトの組み立てを書く
* 毎回補正式を書く

そこで、これらの処理を**関数としてまとめたモジュール**を用意する。これが「**ドライバ**」である。

#### 例：BME280ドライバのインタフェース

```c
// 初期化
void bme280_init(void);

// 温度だけ読む
float bme280_read_temperature_celsius(void);
    /* ここにレジスタ処理が書かれる*/

// 気圧だけ読む
float bme280_read_pressure_hpa(void);
    /* ここにレジスタ処理が書かれる*/

// 温度・気圧・湿度をまとめて読む
void bme280_read_all(float* temp_c, float* press_hpa, float* hum_rh);
    /* ここにレジスタ処理が書かれる*/
```

アプリケーション側は、

```c
float temp;
bme280_init();
temp = bme280_read_temperature_celsius();
```

のように呼び出すだけで良い。内部で

* I2Cによるレジスタ読み書き
* ビットシフトによる生データ組み立て
* 較正値を使った補正式

などが隠蔽されている。
この「複雑なレジスタ操作や補正計算を外側に見せず、わかりやすい関数としてまとめたもの」が**センサドライバ**である。

---

#### 第1章まとめ

* レジスタはCPU内部のものだけでなく、**マイコン内部の周辺回路**や**センサIC内部**にも存在する。
* メモリマップドI/Oにより、特定のアドレスにアクセスすることでGPIOや通信コントローラを制御できる。
* センサ（BME280）の場合、内部レジスタに測定結果が格納されており、ビット操作と補正計算により物理量を得る。
* それら一連の処理をまとめたモジュールが「ドライバ」であり、アプリケーションからは単純なAPIとして扱える。

## 第2章 ペリフェラル

### 2.1 マイコンとは何か

「マイコン（マイクロコントローラ）」は、単なるCPUではない。
1つのチップの中に

* CPUコア
* メモリ（Flash, RAM）
* 各種ペリフェラル（GPIO, I2C, SPI, UART, ADC, タイマなど）

が一体となっている集積回路である。
※ペリフェラルの詳細は後述

```text
┌───────────────────────┐
│       マイコンチップ   │
│ ┌────┐  ┌────────┐    │
│ │CPU │  │ メモリ  │   │
│ └────┘  └────────┘    │
│ ┌─────────────────┐   │
│ │   ペリフェラル群 │   │
│ │ GPIO, I2C, SPI… │   │
│ └─────────────────┘   │
└───────────────────────┘
```

この「ペリフェラル」が、マイコンと外界（センサやアクチュエータ）との橋渡しをする役割を持つ。

---

### 2.2 ソフトウェア的処理とハードウェア的処理

電子工作の処理を、

* **ソフトウェア的に処理する**（CPUがすべてビットをトグルする）
* **ハードウェア的に処理する**（専用回路=ペリフェラルに仕事を任せる）

の二つに分けて考える。

#### ソフトウェア的な処理

例：GPIOを使ってI2Cライクな通信を**自前でビットバンギング(ビットを考えた実装)**する。

メリット：

* どんなピンでも使える
* 理論的には任意のプロトコルをソフトで実装・変更できる（柔軟性）

デメリット：

* CPUが常にビットのON/OFFタイミングを管理するため、負荷が高い
* タイミング精度がクロックや割り込みの影響を受けやすく、高速化に限界がある
* 宇宙空間などの厳しい環境では、ソフトウェア暴走リスクがある

#### ハードウェア的な処理（ペリフェラル）

例：マイコン内蔵のI2Cコントローラに「このアドレスにこれを書け」と命令し、
実際のクロック生成やACK確認はペリフェラルに任せる。

メリット：

* タイミングがハードウェアで保証され、高速で安定して動作する
* CPUは「開始命令」や「完了フラグチェック」など最小限の仕事でよい
* 宇宙機などで重要な「信頼性」「リアルタイム性」が向上しやすい

デメリット：

* ハードウェアは基本的に拡張性がない（一度設計された機能以外には使えない）
* 通信方式ごとに別々のハードウェア（ペリフェラル（I2C, SPI, UARTなど））が必要になる

---

### 2.3 I2C / SPI / UART のソフト実装の大変さ

ここで「ソフトで全部やるとどれだけ大変か」をイメージするため、I2C風のビットバンギングの流れを簡単に示す。

```c
void i2c_bitbang_write_byte(uint8_t data) {
    for (int i = 7; i >= 0; i--) {
        set_sda( (data >> i) & 1 ); // データビットをセット
        pulse_scl();                // クロックをHigh→Lowにする
    }
}
```

実際には、

* スタート条件／ストップ条件の生成
* ACKビットの確認
* 複数デバイスがバスにいる場合の競合処理
* 適切な時間間隔を守るための待ち時間（delay）の挿入

など、かなり多くの処理をCPUが担当することになる。
これを多数のセンサ・複数バス・高速サンプリングで行うのは現実的ではない。

---

### 2.4 通信方式の特徴と「おすすめ」

ここで簡単に3つの代表的なシリアル通信方式を整理しておく。

| 方式   | 線の本数 | 速度   | 典型用途                  | 特徴                     |
| ---- | ---- | ---- | --------------------- | ---------------------- |
| I2C  | 2本   | 中速   | 基板上の複数センサ接続           | アドレスで複数デバイスを共有できる      |
| SPI  | 4本+  | 高速   | 高速ADC, フラッシュメモリ, LCD等 | シンプルかつ高速だがスレーブ数分CS線が必要 |
| UART | 2本   | 中〜低速 | PCとのシリアル通信, ログなど      | ポイントツーポイントでシンプル        |

**BME280のようなセンサを多数つなぐボード内通信**だけを考えると、

* 配線本数が少なく、複数デバイスを1バスにまとめやすい **I2C** が扱いやすいことが多い。
* 一方で、「どうしても高速 & 確実なスループットが必要」な場面では **SPI** が有利である。
* UARTは主に「マイコン ↔ PC」「マイコン ↔ 通信モジュール」など点対点通信で活躍する。

この勉強会では **I2Cを主な例**として扱うと、**レジスタアクセスとアドレス指定**の概念が理解しやすい。

---

### 2.5 マイコン内部の通信ペリフェラル

マイコンは、I2C, SPI, UART などの通信方式ごとに**別々のペリフェラル**を搭載している。

```text
          ┌────────┐
          │  CPU   │
          └──┬───┬─┘
             │   │
      ┌──────┘   └─────────────┐
┌─────────────────┐   ┌────────────────┐
│  I2Cペリフェラル │   │ SPIペリフェラル │  … 等
└─────────────────┘   └────────────────┘
      │                        │
   I2Cピン                   SPIピン
```

それぞれのペリフェラルには**専用のレジスタ群**があり、

* 設定レジスタ（クロック設定、アドレス設定等）
* データレジスタ（送受信バッファ）
* 状態レジスタ（完了フラグ、エラーフラグ）

などが備わっている。

CPUは「I2Cペリフェラルのレジスタに1回書き込めば、残りのビットトグルはハードウェアがやってくれる」。
これが**ハードウェア的処理**の本質である。

---

### 2.6 ペリフェラルとポート（ピン）の関係

重要なポイントとして、

> 「どのペリフェラルが、どのGPIOピンに接続されているか」は**マイコンごとに決まっている**

という事実がある。

* 例えば `I2C1_SCL` は PA9 ピンにしか割り当てられていない
* 同じピンが `GPIO` としても使えたり、`UART_TX` としても使えたりする（ピンの多重機能）

この機能の切り替えは、多くの場合「ピンマルチプレクサ（pinmux）」や「代替機能設定レジスタ」によって行われる。

**デバッグ上の典型的な落とし穴**として、

* 「任意のGPIOピンでI2Cを使えると思い込んでいる」
* 「データシート上で、そのピンにI2Cペリフェラルがつながっていない」

というケースがある。この点は第3章のデバッグでも再び取り上げる。

---

### 2.7 ADCなど他のペリフェラル

ペリフェラルは通信系だけではない。代表的なものとして、

* ADC（Analog-to-Digital Converter）：アナログ電圧をデジタル値に変換する
* タイマ／カウンタ：時間測定や周期割り込みの生成
* PWM（Pulse Width Modulation）：パルス幅制御でモータやLEDを制御する

これらも**マイコンと物理世界との境界で働く回路**であり、
「ペリフェラル」という大きな枠組みで捉えると理解しやすい。

---

#### 第2章まとめ

* マイコンは CPU + メモリ + ペリフェラルが1チップに集約されたデバイスである。
* ソフトウェアだけで通信を実装することもできるが、負荷や信頼性の面で限界がある。
* 通信方式ごとに専用のハードウェア（ペリフェラル）が用意されており、CPUの負荷を大きく減らしてくれる。
* ペリフェラルとGPIOピンの接続関係はマイコン依存であり、データシートを読み解く力が重要である。

---

## 第3章 レジスタやペリフェラルを意識したデバッグ

第3章では、これまでの知識を「実際の開発(デバッグ)」にどう活かすかを扱う。
例として、BME280の値が取得できない場合の解析手順を考える。

---

### 3.1 デバッグの三層構造

トラブルシュートを体系的に行うために、問題を次の三層で考える。

1. **物理層**：電源、配線、信号波形のレベル
2. **プロトコル層**：I2Cのスタート／ストップ、ACK、アドレス等
3. **ソフトウェア層**：レジスタ設定、ドライバのロジック

これを表にまとめると以下のようになる。

| 層      | 主な問題例                 | 使用するツール   |
| ------ | --------------------- | --------- |
| 物理層    | 電源不足、GND未接続、断線        | DMM, オシロ  |
| プロトコル層 | アドレス間違い、ACK無し、クロック不一致 | ロジアナ、オシロ  |
| ソフト層   | 初期化順序ミス、レジスタ値ミス       | デバッガ、ログ出力 |

---

### 3.2 ノミナルな動作の整理

BME280が正常に動作しているときに何が起こるか、まず「理想的なシナリオ」を整理する。

1. マイコンのI2Cペリフェラルが初期化される（クロック設定、ポート設定）。
2. BME280へのI2Cアドレス（例：0x76）が正しく設定される。
3. `ID` レジスタを読んで、期待通りの値（例：0x60）が返ってくる。
4. `CTRL_MEAS` などに測定モードを設定する書き込みが成功する。
5. 一定時間後、`TEMP` レジスタ群から温度の生データが読み出せる。
6. 生データを補正して温度[℃]が得られる。

デバッグの基本は、

> この一連の流れの **どこまで到達できているか** を一つひとつ確認することである。

---

### 3.3 オシロスコープ／ロジアナを使った確認

#### 3.3.1 「そもそも信号が出ているか」を見る

まず、マイコン → センサ方向の信号（SCL, SDA）が物理的に出ているかを確認する。

1. オシロのプローブを SCL ピンに接続し、I2C通信をトリガする。
2. 規則正しいクロック波形が出ているか確認する。
3. 同様に SDA 線にプローブを当て、アドレスやデータの変化が見えているか確認する。

もしここでまったく波形が見えない場合、

* マイコン側のポート設定ミス（I2CピンではないGPIOを使っている）
* I2Cペリフェラルが有効化されていない
* そもそもコードがその箇所まで到達していない

など、**ソフトウェア／設定側の問題**を疑うべきである。

---

#### 3.3.2 センサからの応答を確認する

次に、センサ → マイコン方向の応答を見る。

* I2Cでは、アドレス送信後にスレーブ（BME280）がACKビットを返す。
* その後、読み出しコマンドに応じてセンサがデータをSDAラインに出力する。

ロジックアナライザを用いれば、

* 送っているスレーブアドレスが正しいか
* ACKが返っているか
* 期待するレジスタアドレスが送信されているか
* 読み出したバイト列が、レジスタマップと整合しているか

を簡単に解析できる。

もし「マイコン→センサ方向の波形は正しいが、センサからデータが返ってこない」なら、

* センサへの電源供給（VCC/GND）が不十分
* センサの破損
* アドレスピン設定の違い（0x76と0x77の混同など）

といった**ハードウェア側の問題**を疑う。

---

### 3.4 ペリフェラルとポートのミスに注意

I2CやSPIのデバッグでよくある落とし穴は、

* ソフトウェアでは「I2C1」を使うつもりで初期化している
* しかし実際に配線したピンは「I2C2」のSCL/SDAに対応するピンだった

といった、**ペリフェラルとポートの対応の勘違い**である。

これを防ぐには、

1. マイコンのデータシートやリファレンスマニュアルで、

   * どのピンがどのペリフェラルのどの信号に対応しているかを確認する。
2. ピンマルチプレクサ（Alternate Function）の設定が正しいか確認する。(つまり，使用しているピンがGPIOとして設定されているのか，使用したいペリフェラルに設定されているのかを確認する)
3. テストとして、意図したピンを単純なGPIO出力に切り替え、

   * LED点灯などで「本当にそのピンが制御できているか」を確認する。

---

### 3.5 実践的なデバッグ手順の例（BME280）

BME280で値が読めない場合の、典型的な調査手順をまとめる。
基本的に，レジスタ内部の値が変動しない，対象センサのIDの確認が行われることが多い

1. **物理層の確認**

   * DMMで VCC と GND が正しく配線されているか確認する。
   * オシロで SCL にクロックが出ているかを見る。
   * SDA が Highレベルでプルアップされているか確認する（プルアップ抵抗の有無）。

2. **プロトコル層の確認**

   * ロジアナでスレーブアドレスが正しいか（0x76 / 0x77）確認する。
   * アドレス送信後にACKが返っているか確認する。
   * `ID` レジスタ読み出しが行われていることを波形で確認する。

3. **ソフトウェア層の確認**

   * I2Cペリフェラルの初期化コード（クロック、ピン設定）が正しいか。
   * レジスタマップに基づき、正しいアドレスにアクセスしているか。
   * 生データのビットシフトや補正式が間違っていないか。

このように**層ごとに切り分ける**ことが、効率的なデバッグにつながる。

---

### 3.6 マイコン選定とデータシートを読むという発想

最後に、現場での設計を意識した視点を一つ紹介する。

* どのシリアル通信（I2C, SPI, UART）を何ポート使う必要があるか
* どの程度の速度・精度が必要か
* 何個のセンサ・アクチュエータを接続するか

といった要件に応じて、

* 「I2Cが2チャネル以上必要」
* 「SPIが1チャネルでいいが高速が望ましい」
* 「ADCは12bit以上が必要」

などの**ペリフェラル要件**が決まる。
そのうえで、

* マイコンのデータシートを読み、ペリフェラルの数やピン配置を確認する。
* 選んだマイコンで、目的の接続構成が実現できるかを検証する。

これは、将来的に衛星バス設計やオンボードコンピュータ設計を行う際にも重要な観点である。

---

#### 第3章まとめ

* デバッグでは、物理層・プロトコル層・ソフトウェア層の三つに切り分けて考えると整理しやすい。
* BME280のようなセンサのデバッグでは、

  * マイコン→センサ方向のレジスタ叩き
  * センサ→マイコン方向の応答
    の両方をオシロやロジアナで確認することが重要である。
* ペリフェラルとポートの対応ミスは非常に多い落とし穴であり、データシートを読む習慣が重要である。
* どのペリフェラルを何ポート使うかを事前に考え、それに適したマイコンを選定することが実務につながる。
