# Raspberry Pi PicoとMicroPythonによるモジュール化設計

プログラムの規模が大きくなるにつれ、コードを整理して品質を保つ手法が重要になります。本教材では、関数やクラスを使ったプログラムの分割、モジュール化による開発効率向上、段階的なテスト（ユニットテストと統合テスト）、およびハードウェアとソフトウェアの責任分離による障害追跡といった概念を学びます。

例としてLED点滅、UART通信、PWM制御、タクトスイッチ入力を扱い、コード例は基本的にMicroPython（必要に応じてC/C++の補足）で示します。各セクションでは概念の解説とそれに続く段階的な演習を含み、簡単な模擬バグの事例とその解決法も紹介します。モジュール化設計と段階的テストが組み込み開発にもたらす利点を体感していきましょう。

## 1. 関数によるプログラムの分割
プログラムを整理する第一歩として「`関数（function）`」の活用があります。
MicroPythonにおいて、関数は特定のタスクを実行するステートメントの集まりです。関数を使うことで、プログラムをより小さなモジュール化されたブロックに分割できます。
コードの規模が大きくなるにつれて、関数で処理を分けることでコードが整理され、管理しやすくなります。また、同じ処理を繰り返し書く必要がなくなり重複を避け、一度書いたコードを再利用できるというメリットもあります。

### 関数の基本と利点

関数は`def`キーワードで定義し、名前と必要に応じて引数を指定します。以下に簡単な関数定義の例を示します。

```py
def greet(name):
    print(name + "さん、こんにちは！")
```

上記の`greet`関数は名前を受け取り、挨拶のメッセージを表示します。関数を定義することで、この挨拶処理をプログラム中どこからでも簡単に呼び出すことができます。関数を使う主な利点は次の通りです。

* **コードの整理**：関連する処理を一つのまとまり（関数）として定義し、プログラム全体を見通しやすくします。大きなプログラムでも、小さな部品に分割されていれば理解しやすくなります。
* **再利用性**：一度定義した関数は、他の場所から何度でも呼び出せます。同じコードをコピー＆ペーストする必要がなくなり、修正も一箇所で済みます。
* **保守性**：バグ修正や機能追加の際、関数ごとに処理が分かれていれば影響範囲を限定しやすく、変更やテストが容易です。

## 1.2 LED点滅プログラムで見る関数の活用

それでは具体例として、LEDを点滅させるプログラムで関数の使い方を見てみましょう。Raspberry Pi Picoのボード上には組み込みLED（Pico無印ではGPIO25に接続されたLED）があり、これを点滅（いわゆる「Lチカ」）させることが入門としてよく行われます。 まずは関数を使わずにLEDを点滅させるコードを書いてみます。
```py

import machine
import time

led = machine.Pin(25, machine.Pin.OUT)  # Picoの組み込みLED (GPIO25)
for i in range(10):
    led.value(1)       # LED ON
    time.sleep(0.5)   # 0.5秒間待つ
    led.value(0)       # LED OFF
    time.sleep(0.5)   # 0.5秒間待つ
```
このコードでは、LEDを10回点滅させています。しかし、LEDを点滅させる処理（ON->待機->OFF->待機）が直接プログラム本体に書かれており、もし他の箇所でも同様の点滅処理をしたい場合はコードの重複が発生します。また、点滅の間隔を変更したい場合にも、直接数値を書き換える必要があります。ここで関数を使うことで改善できます。

次に、LED点滅処理を**関数化**してみましょう。


```py
import machine, time

led = machine.Pin(25, machine.Pin.OUT)

def blink_led(pin_obj, times, interval):
    """指定したピンのLEDをtimes回点滅させる関数"""
    for i in range(times):
        pin_obj.value(1)          # LED ON
        time.sleep(interval)     # interval秒待つ
        pin_obj.value(0)          # LED OFF
        time.sleep(interval)     # interval秒待つ

blink_led(led, 10, 0.5)  # LEDを0.5秒間隔で10回点滅
```

関数`blink_led(pin_obj, times, interval)`を定義し、LED点滅のロジックをその中にまとめました。これにより、メイン部分では`blink_led`関数を呼ぶだけで目的の処理が行われます。関数化の利点は以下のように現れています。

* メインのコードが簡潔になり、処理の意図が明確になる（ 「LEDを10回点滅する」と一行で表現できるようになる）。
* 点滅の処理を変更したい場合（例えば点滅回数や間隔）、関数内のコードか呼び出しパラメータを変更すればよく、重複している複数箇所を修正する必要がない。
* 別のピンに接続されたLEDを点滅させる場合でも、`machine.Pin`のオブジェクトを変えて`blink_led`を呼ぶだけで再利用できる。

※「関数を使う」ことを「関数を呼ぶ(call)」と言ったりします

### 演習1:
上記のblink_led関数を応用して、任意の回数・任意の間隔でLEDを点滅させるプログラムを書いてみましょう。
例えば、「1秒間隔で5回点滅した後、0.2秒間隔で5回点滅する」ようなシーケンスを関数呼び出しの組み合わせで実現できます。実行して結果（LEDの点滅パターン）を観察し、期待通りになっているか確認してみてください。

## 2. クラスによるモジュール化と抽象化

関数に慣れてきたら、次のステップとして「 **クラス（class）**」 の概念に触れてみましょう。クラスはオブジェクト指向プログラミングの仕組みで、関連するデータと関数（メソッド）をひとまとめにした入れ物と考えることができます。

クラスを定義すると、その設計図から複数のオブジェクト（インスタンス）を作成でき、各インスタンスはクラスで定義された機能を持ちつつ、それぞれ独立したデータ（状態）を保持します。このようにクラスを使うことで、似たような処理を行う部分を簡潔に表現でき、コードの再利用や拡張が容易になります。

### 2.1 クラスの基本とメリット
Python（MicroPython）におけるクラスの基本構文は以下の通りです。

```py
class クラス名:
    def __init__(self, 引数...):
        # コンストラクタ：インスタンス生成時に呼ばれる初期化処理
        ...

    def メソッド名(self, 引数...):
        # メソッド（インスタンスに属する関数）の定義
        ...
```

`__init__`という特殊なメソッドはコンストラクタと呼ばれ、クラスからオブジェクトを生成するときに自動的に呼び出されます。

ここでインスタンス変数（そのオブジェクト内で保持するデータ）を初期化します。クラス内で定義した関数はメソッドと呼ばれ、第一引数に`self`を取るのがPythonのルールです（`self`はそのメソッドが呼ばれた自身のインスタンスを指します）。クラスを使うことで、データ（属性）とそれを操作する関数（メソッド）を一つにまとめ、外部から見て扱いやすいインターフェースを提供できます。

**クラスを使うメリット:**

* **カプセル化**：データとそれに関わる処理をクラス内に隠蔽できます。外部のコードはクラスが提供するメソッドだけを使えばよく、内部の複雑さを意識せずに済みます。
* **再利用と拡張**：クラスは一種の雛形（ひな形）なので、同じ機能を持つ複数のオブジェクトを簡単に作れます。例えばLEDを扱うクラスを作れば、複数のLEDインスタンスを別々のピンで生成して扱えます。また、クラスを継承して機能を拡張することも可能ですが、初心者のうちは単一のクラスを定義するだけでも十分効果があります。
* **可読性**：クラス名やメソッド名によってコードの意図を表現しやすくなります。プログラムの構造をクラスというまとまりで捉えることで、全体を理解しやすくなります。

### 2.2 クラスを使ったLED制御の例
先ほど関数で実装したLED点滅を、クラスを使って書き直してみましょう。ここではLEDを操作するためのクラス`LEDController`を定義し、初期化でピンをセットアップ、メソッドで点灯・消灯や点滅を行えるようにします。

```py
import machine
import time

class LEDController:
    def __init__(self, pin_no):
        """LED制御クラスのコンストラクタ。pin_noで指定されたGPIOピンを出力に設定"""
        self.pin = machine.Pin(pin_no, machine.Pin.OUT)

    def on(self):
        """LEDを点灯する"""
        self.pin.value(1)

    def off(self):
        """LEDを消灯する"""
        self.pin.value(0)

    def blink(self, times, interval):
        """LEDを所定回数点滅させる"""
        for i in range(times):
            self.on()
            utime.sleep(interval)
            self.off()
            utime.sleep(interval)

# クラスの使用例
led1 = LEDController(25)        # GPIO25に接続されたLEDを制御するインスタンスを生成
led1.blink(5, 0.2)              # 0.2秒間隔で5回点滅

led2 = LEDController(15)        # GPIO15に別のLEDを接続したと仮定
led2.on()                       # led2を点灯
utime.sleep(1)
led2.off()                      # led2を消灯
```
上記のコードで、`LEDController`クラスはLED操作に関する機能をまとめています。`led1`と`led2`のように、同じクラスから複数のインスタンスを作成でき、それぞれ独立したピンに対して動作します。例えば`led1.blink(5, 0.2)`を呼ぶとGPIO25のLEDが点滅し、その間に`led2.on()`でGPIO15のLEDを点灯させる、といった並行した制御も可能です。

クラスを導入することで、コードの意図がさらに明確になります。メインプログラムから見ると、 **「どのGPIOピンのLEDか」を意識せずに、led1やled2といったオブジェクトに対して「点滅せよ」「点灯せよ」と命令している形になります。** ハードウェアの詳細（ピン番号など）はクラス内部に隠蔽され、外部コードは`on()`, `off()`, `blink()`という分かりやすい命令だけ扱えばよいのです。これはまさに**ハードウェア抽象化**の一例であり、モジュール化の恩恵です。

後述するように、ハードウェア依存部分をこのように隔離しておくと、テスト時にソフトウェアロジックとハードウェアの問題を切り分けやすくなります。

## 3. コードのモジュール化と開発効率

**モジュール化**とは、プログラムを機能ごとに分割し、それぞれを独立性の高い部品（モジュール）として設計することです。先の章では関数やクラスによってコードを分割しましたが、さらに大きな観点では**ファイル単位**でモジュールを作成することもあります。

Pythonでは関連する関数やクラスを別ファイルにまとめ、それを`import`文で読み込むことでメインプログラムから利用できます。MicroPythonでも、Pico上にモジュールとなる`.py`ファイルを保存し、例えば`import mymodule`や`from mymodule import MyClass`のようにして機能を分割できます。こうすることで、物理的にもコードが整理され、複数人での開発やコードの再利用が容易になります。

### 3.1 モジュール化の利点

モジュール化された設計には、以下のような利点があります。

* 開発の分業と協調：モジュールごとに担当を分けて開発できます。各人が自分の受け持つモジュール（例：センサ読み取り、通信処理、LED表示など）を実装・テストし、最後に統合するという流れが取りやすくなります。
* デバッグの容易さ：複雑なシステムでも、モジュール単位でテストできれば不具合箇所を切り分けやすくなります。**「複雑なシステムもモジュール化されていれば各部分を個別にテストできるためデバッグが容易になる」**とされており、実際ソフトウェア開発における多くの工数は保守（バグ修正や機能追加）に費やされますが、モジュール構造にしておくことで保守作業も効率化されます。
* 再利用とメンテナンス：汎用的に作ったモジュールは、他のプロジェクトでも使い回せます。また一つのモジュールを改良すれば、それを利用する全ての箇所に効果が及びます。逆に不具合があってもモジュール内に閉じていれば他へ波及しにくく、安心して修正できます。
* 移植性が高い：特にハードウェアに依存する部分（例えばデバイスドライバ的なコード）はモジュールとして切り出し、残りのロジック部分と分離しておくと、将来ハードウェアが変わった際にも他の部分への影響を最小限に抑えられます。これは組み込み開発では重要なポイントで、モジュール化によりハードウェア抽象化レイヤを設けるイメージです。


例えば，同じ階層に次のようにファイルを書き込むようになります．
```
アプリケーション (main.py)
├─ LED制御モジュール (led_module.py)
├─ UART通信モジュール (uart_module.py)
├─ PWM制御モジュール (pwm_module.py)
└─ スイッチ入力モジュール (button_module.py)
```
各モジュールファイルには、そのデバイスや機能に特化したクラスや関数群が定義されています。メインのプログラムはそれらのモジュールを`import`して使用し、高水準のアプリケーションロジック（例えば「ボタンが押されたらLED点滅とUART送信を行う」など）に集中します。

モジュール間の依存は最小限にし、お互いの内部実装に踏み込まないよう **インターフェース（引数やメソッド）** だけを通じてやり取りすることが理想です。<u>**このようなモジュール間の結合が弱い状態（疎結合）を保つことで、変更に強く、バグがプログラムの他の領域に波及しにくいシステムとなります**</u>。

### Tips:

MicroPythonで自作モジュールを作成するには、`.py`ファイルをPico上のストレージに保存します。例えば上記の`led_module.py`に`LEDController`クラスを定義しておけば、メインから`from led_module import LEDController`として読み込み、自由に利用できます。

もしモジュールを読み込めない場合は、ファイル名やパスが正しいか、`.py`の拡張子を付け忘れていないか確認しましょう（Thonny IDEでPicoに保存する際に注意が必要です）。

## 4. 段階的なテスト手法：単体テストと統合テスト

大きなプログラムを一度に書き上げてしまうと、いざ動かしたときにうまく動作しなかった場合、原因を突き止めるのが困難になります。そこで**段階的なテスト**を取り入れましょう。つまり、 **ユニットテスト（単体テスト）と統合テスト（結合テスト）** の考え方です。

* 単体テスト（ユニットテスト）: モジュールや関数など、個々の部品が単独で期待通り動くか確かめる。組み込み開発では、各機能/機器を実機またはシミュレーション環境で個別に試す作業に相当する。
* 統合テスト（結合テスト）: 単体テスト済みの部品同士を組み合わせて、インターフェースの齟齬がないか、全体として正しく機能するかを確認する。<u>**一つずつ部品を増やしながらテストすることで、問題が発生した箇所を絞り込みやすくなる**</u>。

### 4.1 単体テストの実践

単体テストでは、各モジュールについて **専用のテストコード** や **簡易的な確認** を行います。例えば、先ほど作成した`LEDController`クラスの単体テストは非常にシンプルです。Picoに接続されたLEDが正しく点灯・消灯・点滅するか、目視で確認すればよいでしょう。具体的には以下のようなコードを動かしてみます。

```py
# LEDControllerの単体テスト例
led_test = LEDController(25)
led_test.on()
utime.sleep(1)
led_test.off()
utime.sleep(1)
led_test.blink(3, 0.3)
```

これを実行し、LEDが **1秒間点灯 -> 1秒間消灯 -> 0.3秒間隔で3回点滅** というシーケンスで動けば、そのクラスは基本的に正しく機能しています。もし期待通りに動かない場合は、クラス内部（例えば`on()`や`off()`の実装）に問題があるかもしれません。こうした狭い範囲での問題切り分けができるのが単体テストの強みです。

同様に、UART通信モジュールやPWM制御モジュールについても単体テストをします。UARTであれば、所定の文字列を送信する関数を呼び出して実際にPC側で受信できるか確認したり、逆にPCから送ったデータを正しく受信できるか試します。PWMであれば、例えばLEDの明るさを段階的に変えるコードを実行してみて、肉眼で明るさが変化するか確認します。

```py
# PWM制御の単体テスト例（LEDの明るさを徐々に変える）
from machine import Pin, PWM
from time import sleep

pwm_led = PWM(Pin(15))      # GPIO15に接続したLEDをPWM制御
pwm_led.freq(1000)          # 周波数を1kHzに設定
for duty in range(0, 65536, 8192):  # デューティ比を0から100%まで12.5%刻みで変化
    pwm_led.duty_u16(duty)
    sleep(0.5)
pwm_led.deinit()            # PWM停止

```

上記のコードでは、LEDが暗→やや明るい→中程度→...→最大輝度まで段階的に明るくなる様子が観察できるはずです。これでPWM出力自体が機能していることを確かめられます。 タクトスイッチの入力モジュール（例えば`Button`クラス）であれば、スイッチを一定間隔で読み取ってシリアル（UART）に出力するテストが考えられます。

```py
# Buttonクラスの単体テスト例（擬似コード）
button = Button(14)  # GPIO14にスイッチが接続され、内部プルダウンが有効だと仮定
while True:
    state = "PRESSED" if button.is_pressed() else "RELEASED"
    print("Button state:", state)
    utime.sleep(0.2)

```

このようなコードを実行し、タクトスイッチを押したり離したりしながらPCのシリアルコンソール出力を観察します。 **押したときに"PRESSED"、離したら"RELEASED"と表示される** なら単体テストは合格です。もし常に一方しか表示されなかったり反応が不安定であれば、モジュール内の実装やハード接続を疑います。 単体テストを行う際、組み込み環境ではPC上のような高度なテストフレームワーク（例えばpytestなど）を使うことは稀です。むしろ **print文での出力確認やLEDの点灯状態の確認** といったシンプルな方法が主体になります。

MicroPythonでは`print()`関数での出力はUSB経由のシリアルターミナル（REPL）に表示されるため、これをログ代わりに使っても良いでしょう。また、状況によっては **デバッグ用に別のGPIOにLEDやブザーをつないでおき、異常時に光らせる/音を出す** などの工夫も有効です。

例えば「センサー値がしきい値を超えたらLED2を点滅させる」などのコードを入れておけば、PCなしでも簡易的に動作確認ができます。

### 4.2 統合テストの実践

各モジュールの単体テストが済んだら、いよいよそれらを組み合わせてシステム全体の動作を確認します。統合テストでは、いきなり全ての部品を接続するのではなく、「段階的に統合する」ことがポイントです。

*  **2つのモジュールの組み合わせをテスト**: まずはごく一部の組み合わせから始めます。例として、「ボタンを押したらLEDを点滅させる」という統合を試してみます。ButtonモジュールとLEDモジュールを組み合わせ、ボタン押下イベントを検出してLED点滅関数を呼ぶコードを書き、実際にボタンを押してLEDが点滅するか確認します。この段階でうまく動作しなければ、原因はこの2モジュール間のインターフェースや相互作用に絞られます。
* **他のモジュールを追加してテスト**: 次にUARTモジュールも加えて、「ボタンを押したらLED点滅し、かつUARTでメッセージを送る」統合テストをします。LED点滅は既に確認済みなので、追加されたUART出力部分に不具合がないかを中心に観察します。例えば期待するメッセージがシリアルコンソールに表示されるかを確認します。
* **全モジュールの統合テスト**: 最後に、PWM制御なども含めたシナリオ全体をテストします。例えば「ボタンを押すたびにLEDの明るさを段階的に上げ、その度に現在の明度レベルをUART出力する。一定以上の明るさになったらまた0に戻す」といった一連の動作をプログラムし、システム全体として期待通りに動くか確認します。

段階的に統合テストを行うことで、仮に問題が発生しても直前に追加・変更した部分に原因を特定しやすくなります。たとえば、ボタン+LEDではうまくいっていたのにUART出力を加えたら不具合が起きた場合、UARTモジュールやその呼び出しロジックに問題があると推測できます。このように、徐々にシステムを構築しテストすることで、開発途中でのデバッグが効率化されます。

また、統合テストではタイミングやリソース競合にも注意します。組み込みシステムでは、複数の処理が同時進行することで思わぬ不具合（たとえばUART送信中に長時間待ちが発生して他の処理ができない、など）が発生することがあります。モジュール単体では問題なくても、組み合わせるとパフォーマンスが足りなくなるケースもあります。そのため、統合テスト時には各処理のタイミングを調整したり、必要に応じて非同期処理や割り込みを利用する、といった対策も検討します（ただし本教材の範囲では複雑な並行処理は扱いません。必要に応じて簡単なutime.sleep()の値を調整する程度で十分でしょう）。

### 演習4:

以下の統合シナリオを考えて実装・テストしてみましょう。

* **シナリオ例**: 「タクトスイッチを押すごとにLEDの点滅パターンが変わるプログラム」を作る。最初のボタン押下ではLEDがゆっくり点滅、次の押下では速く点滅、その次はPWMで徐々に明るさが変化するフェード点滅、といった具合にモードを切り替える。また、現在のモードをUART経由でコンソールに表示する。
* このシナリオを実現するには、Button入力、LED制御、PWM制御、UART出力のモジュールがすべて必要です。各モジュールの組み合わせを順番にテストしながら最終形に近づけてください（例えば、まずボタンでモードが切り替わるかUART出力で確認→次にLED点滅動作を連動させる→最後にPWMフェードを組み込む、という段階でテストする）。

統合テストでは、すべてが一度にうまくいかなくても落ち着いて対処することが大切です。<u>**一度に直そうとせず、一度動いた段階まで戻して再度確認する、モジュール単体のテストに立ち返る、といった地道な方法で原因を切り分けましょう。**</u>

## 5. デバッグ技法とハードウェア・ソフトウェアの切り分け

組み込みシステムのデバッグでは、「不具合の原因がハードウェアにあるのかソフトウェアにあるのか」をまず見極めることが肝要です。モジュール化設計をしていると、この **責任分離** が比較的容易になります。ここでは、典型的なバグのケーススタディを通じて、効率的なデバッグの流れを紹介します。

### 5.1 デバッグ時の基本テクニック

ソフトウェア的なバグ調査には一般に次のような手法があります。

* **変数の値をprintして確認**: プログラムの途中途中で変数や内部状態をprint()で表示し、想定通りの値かチェックします。MicroPythonではシリアルコンソールに出力されるため、そのログを見てプログラムの流れやデータを追跡します。
* **ログ出力/LEDによるフィードバック**: より体系立てて情報を得るために、printの代わりにログ機能を作ったり、ある条件でLEDを点灯させるなどの方法があります。例えば「ここまでコードが実行されたらLEDを1回点滅させる」「エラーイベントが起きたら赤色LEDを点灯させる」といった具合です。これはシリアル出力が使えない状況や、目視ですぐ確認したい場合に有効です。
* **ステップ実行（ブレークポイント）**: 開発PCと接続してデバッガでステップ実行できる場合は、コードを一行ずつ実行しながら状態を検証する手法もあります。しかし、MicroPythonの場合はPC上で動くソフトとは異なり、標準的なステップデバッガは利用できません（高度な方法としてOpenOCDやpyboardを用いたデバッグがありますが初心者向けではありません）。したがって、本教材では主にprintやLEDを使った手法に頼る形となります。
* **分割統治法（疑わしい箇所の切り分け）**: 前章で触れたように、システムを部分に分けてそれぞれテストすることでバグの潜む場所を絞り込む方法です。プログラムの途中にチェックポイントを作り、「ここまでは正しく動作するが、この先で問題が起きる」という境界を探します。これを繰り返し、問題の出る最小の組み合わせや条件を特定します。

### 5.2 ケーススタディ: ボタン押下でLEDが光らない場合のトラブルシュート

**状況**: 「ボタンを押すとLEDが点灯するはずの回路を組んだが、実際に押してもLEDが光らない。」という問題が発生したとします。ボタンはGPIO14に接続し、LEDはGPIO25（オンボードLED）を使用。ソフトは以下のような単純なものとします。

```py
button = machine.Pin(14, machine.Pin.IN)            # プルダウン/プルアップ設定忘れ
led = machine.Pin(25, machine.Pin.OUT)
while True:
    if button.value() == 1:
        led.value(1)
    else:
        led.value(0)
```
期待される動作は「ボタンを押すとLEDが点灯・離すと消灯」ですが、現実には押しても何も起きないとします。

#### 切り分けのステップ:

1. **ハードウェアとソフトのどちらに問題があるか仮説を立てる**: まず、LEDがつかない原因としては (a) LED側の問題（LEDや出力がおかしい）、(b) ボタン入力側の問題（押しても入力が変化していない）、(c) ソフトの論理の問題、の3つが考えられます。順番に検証しましょう。
2. **LED制御の単独テスト**: LED（GPIO25）の制御が正しく効いているか確認します。例えばREPL上で`machine.Pin(25, machine.Pin.OUT).value(1)`と直接打ち込んでLEDが点灯するか、簡単な点滅スクリプトを走らせてみます。ここでLEDが点滅すれば、LEDとその制御コードには問題ありません。もし点滅しなければ、LEDのハード不良やピン番号の指定ミス、あるいはPicoのボード種類（Pico WではGPIO25ではなく無線モジュール側に乗っているLEDで別の制御が必要、など）の違いといった問題が考えられます。
3. **ボタン入力の単独テスト**: 次に、ボタンの入力値が正しく読めているか確認します。以下のようにボタン状態を継続的に表示するコードを動かします。
```py
button = machine.Pin(14, machine.Pin.IN)
while True:
    print(button.value())
    utime.sleep(0.5)
```
この出力をPCで見て、ボタンを押していないときと押したときで値（0/1）が変化するか確認します。もし**常に0のまま**で変化しなければ、入力が変わっていないことになります。原因として、**内部プルアップ/プルダウンの設定漏れ**が疑われます。タクトスイッチをGPIOとGNDの間に接続した場合、GPIOはプルアップが必要ですし、GPIOと3.3Vの間ならプルダウンが必要です。上のコードではそれを指定していないため、GPIO14は入力フロート状態になり、押しても値が変化しない可能性があります。この場合、修正として例えば以下のようにします。
```py
button = machine.Pin(14, machine.Pin.IN, machine.Pin.PULL_DOWN)
```
こうしてプルダウンを有効にすると、通常時0、押下時に外部から3.3Vが加わって1になるという安定した動作が得られます。

もしボタンを押していないのに**0と1が不規則に出力される**場合も、入力が浮いてノイズを拾っている典型例です。同様にプルアップ/プルダウン設定の問題といえます。

あるいは、ボタンを押しても**常に1のまま**変化しないとしたら、配線が間違っている可能性があります（例えばボタンの両端とも3.3Vに繋いでしまっている、など）。配線図を見直し、マルチメータでGPIO14の電圧が押下で変わるか測る、といったハードウェア的確認も必要です。

上記の検証を行った結果、仮に「GPIO入力にプルダウンを設定したらボタン押下時に値が1になり、LEDも点灯するようになった」としましょう。これは **ハードウェア設定（プルダウン）の問題**が原因であったことになります。モジュール化設計をしていれば、Buttonモジュール内でプルアップ/ダウンを適切に設定していなかったバグとも言えます。このように、 <u>**ハードウェア由来の不具合**と **ソフトウェア上の不具合**を切り分け</u>、一つずつ潰していくことが大切です。

### 5.3 ケーススタディ: 複数回ボタンを押すとLEDがおかしな点滅をする場合

**状況**: 「ボタンを押すごとにLEDが1回点滅するプログラム」を作ったところ、確かに点滅はするものの、素早くボタンを連打するとLEDが意図しない点滅をしたり、1回しか押していないのに2回点滅することがある、という問題が起きたとします。これは組み込み開発で頻出する**チャタリング（スイッチの接点のバウンス）**による症状です。機械的なスイッチは押した瞬間に接点が何度もバタつくため、数ミリ秒の間に押したり離したりを高速で繰り返したような信号がGPIOに入力されます。その結果、ソフト的にはボタンを1回押したつもりでも複数回押されたように見えてしまいます。

**対策とデバッグ:**

1. **現象の観測**: UARTのログやLEDの追加点滅などで、ボタン押下イベントが何回認識されているか確認します。例えばボタンが押されるたびにカウンタを+1してUARTに出力するようなコードを仕込んでみるとよいでしょう。一度の押下でカウンタが2以上増えていれば、チャタリングが発生していると判断できます。

2. **簡易的なデバウンス:** チャタリング対策（デバウンス）としては、**一定時間入力を無視する**方法が簡単です。例えば、ボタンが押されたらその後50ms程度は追加の押下を受け付けないようにする処理を入れます。具体的には以下のようなロジックになります。
```py
if button.value() == 1 and not prev_state:
    # ボタンが離されている状態から押された状態に変化した
    led.blink(1, 0.2)
    utime.sleep(0.05)  # 50ms待機（この間のバウンス無視）
prev_state = button.value()
```
このように、押下を検出したら少し待つことで、バウンスによる2回目以降の押下信号を拾いにくくします。高度な方法ではRCフィルタやソフトウェアでの時系列判定を行いますが、初心者レベルでは上記のような**簡易ディレイ**(インターバルを設けること)でも効果はあります。

3. 再テスト: デバウンス処理を追加後、再びボタンを何度か押してみて現象が改善するか確認します。チャタリングによる誤動作がおさまっていればOKです。

このケーススタディでは、最初はハード的な要因（スイッチの物理的特性）が原因でしたが、それをソフトウェア側で対処することで問題を解決しました。組み込みシステムでは、このようにハードとソフトの境界にまたがる問題も多々あります。<u>**ハードウェアの制約や現象を十分に理解しつつ、ソフトウェアでどこまでカバーするか**</u>を考えるのも組み込みエンジニアの腕の見せ所です。

### 5.4 その他のデバッグヒント

* **ハードウェアの独立テスト**: ソフトに入る前に、疑わしいハード部品は単体でテストすることも有効です。マルチメータやオシロスコープが使えるなら、センサー出力電圧や通信ラインの信号を直接観測してみましょう。例えば「センサー値が常に0になる」バグでは、実はセンサーが壊れていた、配線が間違っていたということもあります。ハードが想定通り動作していることを確認できれば、ソフトの問題に集中できます。
* **ソフトウェアのスタブ化**: 逆にハード依存の部分をソフトウェアでエミュレート（スタブ）してテストする手法もあります。例えばネットワーク通信モジュールを開発中で、まだ実機サーバに接続できない場合、通信部分をスタブに置き換えて決まったデータを返すようにし、上位ロジックのテストを進めることがあります。MicroPythonでも、クラスの差し替えや条件分岐で疑似データを返すようにして、センサーがない状態でもプログラムを動かしてみるといった工夫が可能です。
* **ドキュメントの参照**: 組み込みのバグは、マイコンやセンサの仕様をよく読むと思わぬヒントが隠れていることがあります。たとえばRP2040マイコン（PicoのCPU）のPWMは複数ピンでスライスを共有する仕様があります。もし「別々のPWM出力が干渉する」といった現象が起きたら、データシートやリファレンスを確認してハード仕様由来の動きかどうか疑ってみるべきです。
* **コミュニティの活用**: 自分で行き詰まったら、Raspberry Pi公式フォーラムや技術ブログ、Qiita記事などコミュニティリソースを調べてみましょう。「Pico UART 動かない」「MicroPython PWM not working」などで検索すると、類似の問題と解決策が共有されていることがよくあります。
